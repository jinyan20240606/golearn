# 03并发编程

1. 概念
   1. 进程 = 独立工厂资源隔离，占地大，搬家（切换）极慢。
      1. 进程：操作系统资源分配最小单位，完全独立内存空间
      2. 线程：进程内执行单元，共享进程内存，OS 内核调度
   2. 线程 = 工厂里的工人一个工厂多个工人，工人换岗需要领导（操作系统）调度，成本高。
   3. 协程 = 同一个工人「多任务穿插干活」
2. 协程概念：同一根线程里面，人为分出多个「独立的代码执行流」，可以手动 / 自动 挂起、暂停、恢复，在单线程内实现并发，切换开销几乎为 0，
   1. 跑在单个线程里
   2. 多段代码交替执行
   3. 切换不用内核、不用系统调用
   4. 开销极小、内存极小、几万 / 几十万随便开

## 传统语言和方案演进

其他语言中的并发编程一般是通过多线程或多进程来实现，操作系统级别上：多线程或多进程存在的问题，很多时候如web2.0场景力不从心
1. 耗费内存，内存占用很大
2. 线程切换的开销大


Go语言中，并发编程是通过协程（Goroutine）来实现的。很多语言也都有协程，如python的asyncio，php-swoole，java的虚拟线程是协程netty，C++的libuv，Rust的tokio，**Go语言的Goroutine**。go语言中没有提供线程的，因为他是web2.0后的新语言，一开始设计时就没有设计线程，所有的库都是协程级，没有线程级。

1. 线程级的核心痛点（Web2.0 高并发场景致命短板）
   1. 内存占用极高
      1. 单个 OS 线程默认栈内存通常 1MB～8MB
      2. 上万并发连接 = 上万线程，内存直接爆炸，服务器上限极低
   2. 上下文切换开销大
      1. 线程切换需要：内核态 / 用户态切换、寄存器保存、页表刷新、全局资源调度
      2. 高并发下，大量 CPU 资源浪费在切换而非业务计算
   3. 调度由内核全权接管
      1. 内核调度粒度粗，无法针对业务轻量任务优化
      2. 海量阻塞线程会拖慢整体调度效率
   4. 开发成本高
      1. 需手动控制锁、竞争、死锁、上下文隔离
      2. 线程池容量固定，伸缩性差
----

- 几乎所有主流语言，都是 线程 + 协程 同时支持，只有 Go 语言是特例：完全不暴露系统线程，只给你用 Goroutine（协程）
- nodejs中也没有真正意义上的协程，只是伪协程，只有异步事件循环，nodejs支持高并发，是因为单线程+非阻塞IO=能抗大量链接，但它不是靠协程，是靠事件驱动
  - 因为 async/await 写法长得像协程，但底层完全不是一回事。
- 协程（Goroutine / PHP 协程 / Python 协程）的特点：
  - 用户态切换
  - 可以挂起 / 恢复
  - 多个执行流并发跑在单线程里
  - 有独立栈、独立上下文
- Node.js 的 Promise /async-await 特点：
  - 还是单线程
  - 还是事件循环
  - 还是回调包装
  - 没有栈切换、没有上下文保存、没有真正挂起

## go语言中的协程

- 协程（goroutine）是Go语言中的一种轻量级线程，它与线程相比，具有以下特点：
  - 主要是：内存占用小，切换快（因为不是操作系统线程级切换，而是函数式切换），执行效率高
  - 依附于线程，用户态调度
    - 不经过操作系统内核，语言 / 框架自己调度，切换极快。
  - 可挂起、可暂停、可恢复
    - 遇到 IO（数据库、网络、等待）主动让出执行权，不卡死线程。
  - 轻量极致
    - 系统线程：默认栈 1MB～8MB
    - 协程：栈 2KB～几 KB，单机轻松十万级并发

1. 主流语言协程实现
   1. Python：asyncio 单线程异步协程
   2. Java：Netty 基于 Reactor 事件循环 + 异步模型
   3. C++/PHP：libuv、Swoole 协程
   4. Rust：tokio 异步运行时
   5. Node.js：底层 libuv 事件驱动的asyncawait等伪协程
   6. Go：Goroutine 原生语言级协程
2. 协程通用优势
   1. 用户态调度：不陷入内核，切换开销极小
   2. 极小内存：初始栈内存 KB 级别，支持动态伸缩
   3. 轻量化：单机轻松支撑十万、百万级并发
3. goroutine的特点
   1. 无原生线程模型，全链路协程化
      1. Go 诞生于 Web2.0 高并发时代，设计之初就摒弃传统线程模型
      2. 标准库、网络库、IO 操作、定时器全部基于 Goroutine 设计
      3. 没有暴露系统线程 API，业务层只能使用 Goroutine
   2. GMP 自研调度器（Go 运行时级别）
      1. G：Goroutine（轻量任务）
      2. M：OS 系统线程
      3. P：逻辑处理器（调度上下文）
      4. 由 Go 运行时 自主调度，而非操作系统内核
      5. 多路复用：少量 OS 线程 承载 海量 Goroutine
   3. 极致低开销
      1. Goroutine 初始栈仅 2KB，运行时动态扩容 / 缩容
      2. 上下文切换仅需保存少量寄存器，无内核态切换
      3. 单机轻松开启 百万级 Goroutine
   4. 配套原生并发能力
      1. channel 实现协程通信，共享内存不用乱加锁
      2. sync 基础原语、CSP 并发模型
      3. 网络 IO 自动多路复用、阻塞协程自动调度挂起

## 3-4 通过mutex和atomic完成全局变量的原子操作

使用多协程并发工作时，每个协程的函数逻辑必须是原子性操作，因为多个协程并发执行时，可能会产生数据不一致的情况，所以需要使用锁机制来保证数据一致性

### 举例子

1. 你以为 a += 1 是：`a = a + 1`，但CPU 真正执行是 3 步：
   1. 读取 a（从内存拿到当前值）
   2. 计算 a+1
   3. 写回内存
   4. 这三步之间，别的协程可以插进来！于是就会出现数据错乱。
2. 经典冲突例子
   假设：
   协程 1：a += 1
   协程 2：a -= 1
   初始 a = 0
   理想结果应该是：0 +1 -1 = 0
   但真实并发可能变成：
   协程 1 读取 a = 0
   协程 2 读取 a = 0
   协程 1 计算 0+1=1
   协程 2 计算 0-1=-1
   协程 1 写回 a=1
   协程 2 写回 a=-1
   最终结果变成 -1，完全错误！
   这就叫：**非原子操作导致的数据竞争（race condition）数据不一致**

### 解决

保证原子性：就是并发执行时，不要插队交叉执行，要等上一个协程执行完毕，再执行下一个协程。

1. 使用sync.Mutex互斥锁机制`var lock sync.Mutex`
   1. 相当于给竞争变量加锁，只有获取锁的协程才能执行，其他协程只能等待，变成串行了
2. 很多时候我们可以使用atomic包中的原子操作函数
   1. atomic包中的原子操作函数都是原子操作，不会产生数据竞争
      1. 原生只支持以下类型：数值 + 指针
         1. nt32，int64，uint32，uint64，uintptr，unsafe.Pointer  // 指针
   2. 只要你所有地方都统一用 atomic 操作同一个变量
   3. 混用普通操作 + atomic 一定会数据竞争
   4. 适合：
      1. 简单的数值读写（int32/int64/uint32/uint64/uintptr）
      2. 标志位、状态、计数器、序列号
      3. 极高的场景
   5. 不适合：
      1. 复杂结构体（建议用 Mutex）
      2. 多条逻辑需要捆绑执行（必须用 Mutex）
   6. 原理：atomic 完全不用锁（Mutex）！它靠的是 CPU 硬件指令 实现原子性！
      1. 锁 = 软件控制atomic = 硬件直接保证
      2. CPU 有一条指令：原子执行（不可中断），这些指令的特点：整个执行过程中，其他 CPU 核心不能插手


## 3-5 RWMutex读写锁

前面的Mutex是互斥锁，它只能让一个协程执行，其他协程只能等待，变成串行了。这种肯定影响性能

但是，有时候我们希望多个协程可以同时读，但只有一个协程可以写。

比如：数据库的连接池，多个协程可以同时读取连接，但只有一个协程可以写入连接。（就是读协程之间可以并发执行，读和写之间应该互斥串行即写的时候你也不能读）

这时，就需要读写锁了。

- 我们宗旨是：锁不得不用时再用，设计锁时尽量保证并行，串行性能太差了

## 3-6 通过channel实现goroutine之间的通信

> go语言经典话：不要通过共享内存来通信，而是通过通信来共享内存

- 线程之间的通信一般两种
  - 共享内存方式：php,python,java中，线程之间通信,最简单常用的方式是在内存中声明一个全局变量，多个线程之间查询全局变量的情况进行读写
  - 消息队列方式：的机制，如python-queue，java即 消费者和生产者之间的关系
- go语言中的channel也是基于消息队列原理，通过channel的发送和接收操作，实现goroutine之间的通信，channel加上语法糖让使用更简单

### channel大概原理

就是两个线程之间搭起一个通道，两个线程之间通过通道进行通信，- 一个线程往通道中发送数据，另一个线程从通道中接收数据
- 通道中可以存放多个数据，当通道中存放的数据达到上限时，发送方会阻塞，直到通道中数据释放，接收方会阻塞，直到通道中有数据
- 通道中的传递的数据一般需要定义类型，可以是基本类型，也可以是自定义类型
- channel的底层是通过一个环形的数组来实现的，

![alt text](image-1.png)

1. channel的初始化：容量为0就是无缓冲channel，容量大于0就是有缓冲channel


### happen-before机制

它规定了不同 goroutine 间内存操作的可见性与执行顺序保证

解释了channel/main文件中main1方法对于容量为0的channel为什么必须开协程 + 为什么这就不死锁了？？

一、机制

1. 无缓冲 channel 必须：接收者先运行，发送者才能发送
2. 有缓冲 channel（容量 ≥1）= 可以直接先发，不用等

二、Go 给 channel 定的 Happen-Before 铁律

对 无缓冲 channel（容量 0）：
1. 接收操作 happen-before 发送操作完成
大白话：发送者必须有人先等着接收，你才能发送成功！
2. 发送不完成，代码就不会往下走（阻塞）
主协程会卡住，直到有人接收。

三、无缓冲时为什么开协程就能解决？

因为 go 关键字启动协程 是立刻异步执行的，发送时，发现有人已经在协程中等待接收 → 直接发送成功

## 3-9 单向channel的应用场景

1. channel默认是双向的，但是，我们经常一个channel作为参数进行传递，希望对方是单向使用的，这时，单向channel就派上用场了
   1. 如`data := <-msg`中接收读取数据时，其实此时也能继续往里写数据，这样其实就不好，作为一个消费者有时候不希望你能有写的权限，只想让你读
2. 语法
   1. 前提：必须先建一个双向channel，然后根据双向channel创建单向channel，不能直接创建单向channel
   2. 定义单向channel的写法：符号在双向channel的位置-左侧只读，右侧只写，不带符号就是双向
