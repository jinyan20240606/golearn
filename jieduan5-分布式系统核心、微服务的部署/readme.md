#### 1-2 数据库事务的ACID特性

**一、普通事务（如本地事务）**：
1. 是什么
一组 SQL 打包成一个整体要么全部成功，要么全部失败回滚。
2. 作用
   1. 失败了能回滚
   2. 中间状态别人看不见
   3. 保证数据不出错
3. 事务的标准是ACID（必考）
   1. 原子性：功能不可再分，要么全成，要么全不成
   2. 一致性：操作前后数据合法、完整
      1. 事务执行全程，数据始终保持合法有效状态，中间过渡状态对外不可见，只会存在初始态与最终合法态。
      2. 转账实例拆解
         1. 初始合法态：双方金额无变动
         2. 非法中间态：张三扣款、李四未入账，账面总额失衡
         3. 最终合法态：扣款入账全部完成
      3. 一致性约束效果：其他事务查询时，只能读到状态 1 或状态 3，无法读取到状态 2，规避账面数据错乱。
   3. 隔离性：事务之间互不干扰
      1. 并发事务彼此互不干扰，操作数据相互隔离，不会互相篡改影响。
      2. 并发三类问题
         1. 脏读：读到其他事务未提交的临时数据
         2. 不可重复读：同一事务内，两次查同一条数据结果不一样
         3. 幻读：同一事务内，两次查询数据条数出现增减
      3. 例如数据库mysql的事务隔离级别：并发会产生脏读、不可重复读、幻读三类问题，mysql的四个隔离级别，逐级加高约束、逐步解决问题，代价是并发性能逐步下降。
   4. 持久性：是事务的保证，一旦提交，永久生效，事务提交完成后，数据永久写入磁盘，断电、故障都不会丢失。
4. 转账示例中ACID体现：
   1. 原子性：保障扣款、入账要么全做、要么全撤销
   2. 一致性：保障过程数据合法，异常状态不对外暴露

**二、分布式事务**：
1. 是什么
   1. 一次业务操作，跨了多个数据库、多个服务、MySQL+ES、订单 + 库存 + 支付需要所有服务一起成功 / 一起失败。
   2. 分布式事务顾名思义就是要在分布式系统中实现事务，它其实是由多个本地事务组合而成。
   3. 对于分布式事务而言几乎满足不了ACID，其实对于单机事务而言大部分情况下也没有满足ACID，不然怎么会有四种隔离级别呢?所以更别说分布在不同数据库或者不同应用上的分布式事务了，
2. 为什么需要
   1. MySQL 事务管不到 ES订单服务事务管不到库存服务不保证会出现：钱扣了、库存没扣、订单没生成
3. 解决什么问题
   1. 跨服务、跨数据库、跨数据源的数据一致性比如：
   2. MySQL + ES
   3. 订单 + 库存 + 支付
   4. 微服务之间的数据统一

##### mysql隔离级别概念补充

隔离级别属于数据库事务的特性，它决定了事务的隔离级别，即事务之间如何隔离，以保证数据的一致性。

- 隔离级别的产生原因：事务内修改只在自身会话生效，未提交时，别的会话能不能看见，由隔离级别决定
- 并发会产生脏读、不可重复读、幻读三类问题
  - 不可重复读
    - 同一事务先后两次查单条数据，中途别的事务提交修改，两次查到数值不一样。
  - 幻读
    - 同一事务先后两次按条件查一批数据，中途别的事务增删并提交，查到的数据条数变了。
  - 脏读
    - 事务读到另一事务还没提交的修改数据。

- mysql事务过程中的隔离级别：并发会产生脏读、不可重复读、幻读三类问题，四个隔离级别，逐级加高约束、逐步解决问题，代价是并发性能逐步下降。
   1. 读未提交：可以读到别人事务还未提交的数据
      1. 存在问题：脏读、**不可重复读**、幻读 全都有
      2. 意思：能读到别人事务没提交的数据
   2. 读已提交：
      1. 问题：不可重复读、幻读
      2. 意思：只能读到别人事务已提交的数据（解决脏读）
   3. **可重复读**
      1. MySQL 默认级别
      2. 问题：幻读（理论上有，实际上靠间隙锁缓解）
      3. 意思：事务A修改数据未提交，其他事务查询，**依旧读到修改前旧数据**，不会脏读。
   4. 串行化：不支持并发
      1. 问题：无任何问题
      2. 缺点：完全不支持并发，性能极差
      3. 意思：事务排队执行，像单线程一样

   1. 没有发发送出去
   2. 发送出去了，返回没有收到，导致以为出错了
2. 程序出错
   1. 代码异常
   2. 宕机
   3. 断电
   4. 系统问题·磁盘满了、电脑坏掉等

- 网络抖动
  - 短时间延迟忽高忽低、偶尔丢包
  - 信号干扰：无线穿墙、电磁干扰、周边设备信号争抢
  - 链路切换：路由自动换转发线路，瞬时波动
  - 设备调度：路由器网卡间歇休眠、算力波动
- 网络拥塞
  - 通路数据太多挤住，传输变慢超时
  - 带宽上限固定，同时大量请求扎堆涌入
  - 交换机 / 服务器处理能力饱和，数据包排队积压
  - 跨运营商、跨地域转发，中转节点流量爆满

拥塞：早晚高峰马路，车流堵着走不动

### 2章 cap理论和分布式事务解决方案

#### 2-1 CAP理论
cap理论是分布式系统的理论基石

1. CAP 理论精简速记
   1. 三大特性
      1. C 一致性：数据更新后，所有节点同一时刻数据完全相同
         1. 更新操作成功井返回客户端后，所有节点在同一时间的数据完全一致，这就是分布式的一致性。一致性的问题在并发系统中不可避免，对于客户端来说，一致性指的是并发访问时更新过的数据如何获取的问题。从服务端来看，则是更新如何复制分布到整个系统，以保证数据最终一致。
         2. 节点之间有个同步时间即核心是存在中间状态，中间状态一旦暴露给用户访问，则会导致数据不一致，解法可以用分布式锁
      2. A 可用性：服务随时可访问，请求能正常响应，无超时失败
         1. 要保证数据一致，就得加锁阻塞等待数据全网同步，期间无法正常受理新请求，可用性下降，但是要保证可用性随时服务，就得保证随时能访问，所以这C和A是互斥的
      3. P 分区容错性：节点间网络断开、分区故障，系统仍可正常运行
         1. 比如consul系统，她就是分布式有多个系统，他们之间若一个出现问题，对外提供的统一接口还是仍然可用
         2. **正常分布式系统，必须满足CP特性**，但是CAP这3点不能同时满足，意味着C和A只能取舍满足
   2. 核心结论
      1. 分布式系统无法同时满足 CAP 三者，P必须满足，意味着C和A只能取舍满足
      2. CP：舍弃可用性，保证数据一致、容忍分区故障，它首先保证的是同步完数据之后你才能访问
         1. 例如：NoSql，MongoDB，HBase，redis
      3. AP：舍弃强一致，保证服务可用、容忍分区故障
         1. 例如：Mysql，Oracle
      4. CA：分布式场景无法实现，网络必然存在分区风险，仅单机数据库无跨节点网络问题，不用考虑分区，不属于分布式范畴


#### 2-2 Base理论

分布式系统中的一致性是弱一致性（它不可能做到强一致性），单体数据库如myslql的一致性是强一致性

BASE是Basically Available (基本可用)、Soft state (软状态) 和Eventually consistent (最终一致性性) 三个短语的9缩写，8ASE理论是对CAP中一致性和可用性权衡的结果，

其来源于对大规模互联网系统分布式实践的总结，是基于CAP定理逐步演化而来的。

8ASE理论的**核心思想**是:即使无法做到强一致性，就把一致性和可用性各退一步，基本达到要求即可，但每个应用都可以根据自身业务特点，采用适当的方式来使系统达到最终一致性。接下来看一下BASE中的三要素:

1. BASE 理论：是 CAP 取舍后的落地设计思想，适配高并发分布式业务
   1. Basically Available 基本可用
      1. 故障时保留核心功能，非核心功能降级，不会彻底瘫痪，绝不等价于系统不可用
      2. 比如:
         1. (1)响应时间上的损失。正常情况下，一个在线搜索引擎需要在0.5秒之内返回给用户相应的查询结果，但由于出现故障，查询结果的响应时间增加了1~2秒
         2. (2)系统功能上的损失:正常情况下，在一个电子商务网站上进行购物的时候，消费者几乎能够顺利完成每一笔订单，但是在一些节日大促购物高峰的时候，由于消费者的购物行为激增，为了保护购物系统的稳定性，部分消费者可能会被引导到一个降级页面
   2. Soft State 软状态
      1. 节点数据允许短暂不一致，存在中间过渡状态
   3. Eventually Consistent 最终一致性
      1. 不追求实时一致，等待数据同步完成后，最终全部统一
2. 实际淘宝中的例子：
   1. 基本可用：双十一大促或故障时，系统不崩，下单付款正常，慢点、少点功能没关系。
      1. 变慢一点：平时搜商品 0.5 秒，双十一可能 1–2 秒才出来，但能搜到。
      2. 部分功能暂时没了：
         1. 高峰期暂时不让你看历史订单、评价、积分详情；
            1. 甚至弹出 “系统繁忙，请稍后再试” 的降级页，但下单、付款还能走。
      3. 核心保住：浏览、加购、下单、支付 ——不能断；次要功能可以暂时关掉。
   2. 软状态：下单了库存没立刻减、付完钱显示处理中，业务中用户看到提示也可以接受并不强希望立即完成 —— 短暂不一致。
   3. 最终一致：显示等待一段时间后，库存、订单、支付状态全部自动对齐。
3. 两者关系
   1. CAP 是理论取舍准则，BASE 是实际业务妥协方案，电商、支付大多遵循 BASE
   2. 一句话BASE才是你最终业务使用践行的归宿，CAP就是告诉你:想要满足C、A、P就是做梦，



#### 2-3 分布式事务-2阶段提交2PC

- 分布式事务基本首要解决问题就是：同时成功或同时失败，这都解决不了那一定不算解决。
- 常见分布式事务解决方案
  - 两阶段提交(2PC, Two-phase Commit)
  - TCC补偿模式
  - 基于本地消息表实现最终一致性
  - 最大努力通知
  - 基于可靠消息最终一致性方案

我们项目中最终选择基于可靠消息最终一致方案，其他方案选择大概讲讲原理即可

**2PC大致原理**：

- 2PC核心定义：把分布式事务拆成准备、提交两个阶段，由协调者统一管控所有参与节点，保证要么全部成功、要么全部回滚。
  - **主要利用数据库自己的事务能力**
- 角色划分：
  - 协调者：统筹全局，下发指令、汇总结果
  - 参与者：各个数据库 / 服务节点，执行业务数据操作
- 阶段一：准备阶段（Prepare）
  - 协调者向所有参与者下发预执行指令
  - 各参与者执行事务操作，锁定资源，暂不正式提交，**主要利用数据库自己的事务能力**
  - 执行无误则回复就绪；执行异常直接反馈失败
- 阶段二：提交 / 回滚阶段
  - 全部参与者都返回就绪
  - 协调者下发提交指令，所有节点正式提交事务，释放资源
  - 任意参与者报错 / 超时无响应
  - 协调者下发回滚指令，所有节点撤销操作，数据恢复原状
- 实操举例（跨库转账）
  - 参与者：A 账户库、B 账户库
  - 准备阶段协调者通知两个库，分别执行扣款、增款操作，锁住数据，不提交，两个库都反馈执行完成
  - 提交阶段协调者确认全员就绪，下令正式提交，双方账目更新生效
  - 中途任一库执行失败，立刻全员回滚，钱款恢复初始状态
- 优缺点
  - ✅ 优点：强数据一致性，逻辑简单标准，实现起来简单
  - ❌ 缺点
    - ❌ 只要我使用了数据库的事务能力，就一定会有锁一定有性能影响，全程锁定资源，并发性能低
      - 加锁这是最重要的缺点，导致业务很少用
    - 协调者宕机，所有参与者卡住不动，事务卡死无法推进
    - 网络超时容易引发数据异常
    - 一旦某个服务发生故障，则整个事务都会卡住
- 适用场景：业务里极少使用，适用低并发


#### 2-4&5 TCC分布式事务实现方案

TCC = Try - Confirm - Cancel就是把一个分布式事务拆成 3 步，自己手动控制，比 2PC 灵活、性能高很多。

- 先预留资源，没问题就确认提交，有问题就撤销回滚。**全程不锁库用业务逻辑层实现锁的效果，性能比 2PC 强太多**！

3 个阶段拆开讲（超级易懂）

1. Try（预留 / 检查）
   1. 做两件事：检查资源 + 预留资源
   2. 检查：库存够不够、钱够不够
   3. 预留：把要用的资源先 “占住”，不真正扣减
   4. 例子：订单要扣 1 件库存 → Try 阶段冻结 1 件库存，不是直接删掉。
2. Confirm（确认提交）
   1. 所有 Try 都成功 → 执行真正提交
   2. 冻结的库存真正扣掉
   3. 冻结的金额真正扣除
   4. 数据正式生效
3. Cancel（取消回滚）
   1. 任意一个 Try 失败 → 全部撤销
   2. 解冻库存
   3. 解冻金额
   4. 数据回到最初状态
4. 生活例子：淘宝下单（最直观）
   1. Try
      1. 你下单 → 系统冻结商品库存、冻结优惠券
   2. Confirm
      1. 你付款成功 → 库存真正扣减、优惠券真正使用
   3. Cancel
      1. 你取消订单 / 超时未付 → 库存解冻、优惠券退回
   4. 这就是 TCC！
5. TCC 优点（为什么不用 2PC 用 TCC）
   1. 不锁数据库，并发性能极高
   2. 自己控制逻辑，灵活度高
   3. 不会像 2PC 那样卡死、阻塞
   4. 互联网公司最常用分布式事务方案
6. TCC 缺点
   1. 代码量多，要写 3 套逻辑（Try/Confirm/Cancel）
   2. 要自己处理幂等、悬挂、空回滚等问题
   3. 实现TCC这3个接口比较简单，但是在调用方保证协调这几个接口不出错还是比较麻烦的，因为只要是网络调用都有可能出问题

#### 2-6 TCC分布式事务实现方案的实现难点



总结一下，你要玩儿TCC分布式事务的话,完整的解决方案还是很复杂的，要实现事务管理器，最好借助开源框架:
1. 你要自己实现 TCC，必须搞定下面全部难题
   1. 你必须写一个 事务管理器（核心）它要干这些事：
      1. 记录所有参与 TCC 的服务
      2. 记录全局事务状态
      3. 超时了自动触发 Cancel
      4. 宕机重启后恢复事务
      5. 保证 Confirm/Cancel 必须执行成功（最终一致性）
      6. 没有事务管理器 = TCC 直接废了
   2. 必须解决 3 个恶心问题（必踩坑）
      1. ① 空回滚：Cancel 先执行了，Try 还没执行 → 不能报错，要直接忽略
      2. ② 幂等：Confirm/Cancel 可能被重试多次 → 必须保证执行一次和多次结果一样
      3. ③ 悬挂：Try 执行慢，Cancel 已经先回滚完了 → Try 不能再执行
      4. 这三个问题不处理，TCC 数据必乱
   3. 网络超时、服务宕机，你都要兜底
      1. Try 成功了，但服务挂了 → 必须触发 Cancel
      2. Confirm 发出去没响应 → 必须重试
      3. 整个分布式系统断网 → 事务不能丢
2. 首先需要选择某种TCC分布式事务框架，各个服务里会有这个TCC分布式事务框架在运行。
3. 改造成本：你原本的一个接口，要改造为3个逻辑，Try-Confirm-Cancel。
   1. 先是服务调用链路依次执行Try逻辑
   2. 如果都正常的话，TCC分布式事务框架推进执行Confirm逻辑，完成整个事务
   3. 如果某个服务的Try逻辑有问题，TCC分布式事务框架感知到之后就会推进执行各个服务的Cancel逻辑，撤销之前执行的各种操作。
   4. 这就是所谓的TCC分布式事务。

- TCC框架能力实现的终极大招
  - 如果有一些意外的情况发生了，比如说订单服务突然挂了，然后再次重启，TCC分布式事务框架是如何保证之前没执行完的分布式事务继续执行的呢?
    - TCC事务框架都是要记录一些分布式事务的活动日志的，可以在磁盘上的日志文件里记录，也可以在数据库里记录。保存下来分布式事务运行的各个阶段和状态。
  - 万一某个服务的Cancel或者Confirm逻辑执行一直失败怎么办呢?
  - 那也很简单，TCC事务框架会通过活动日志记录各个服务的状态。
  - 举个例子，比如发现某个服务的Cancel或者Confirm一直没成功，会不停的重试调用他的Cancel或者Confirm逻辑，务必要他成功!
    - 当然了，如果你的代码没有写什么bug，有充足的测试，而且Try阶段都基本尝试了一下，那么其实一般Confirm、Cancel都是可以成功的!
  - 当有面试官刁难问，如果一直失败咋办？
    - 首先框架会基于事务日志无限重试 Confirm 或 Cancel 流程，应对网络抖动、临时服务卡顿这类常见故障，绝大多数情况都能重试成功。若持续重试依旧失败，说明遇到数据库损坏、硬件故障、数据错乱等极端异常，系统会触发邮件、短信告警通知运维人员。技术层面不存在百分百绝对成功的事务方案，此时只能人工核对数据、手动修正业务状态，兜底保障最终数据一致。

##### TCC优缺点
**优点:**
1. 解决了跨服务的业务操作原子性问题，例如组合支付，订单减库存等场景非常实用
2. TCC的本质原理是把数据库的二阶段提交上升到微服务来实现，从而避免了数据库2阶段中锁冲突的长事务低性能风险。
3. TCC异步高性能，它采用了try先检查，然后异步实现confirm，真正提交的是在confirm方法中;
**缺点:**
1. 对微服务的侵入性强，微服务的每个事务都必须实现try，confirm，cancel等3个方法，开发成本高，今后维护改造的成本也高。
2. 为了达到事务的一致性要求，try，confirm、cancel接口必须实现等幂性操作。(定时器+重试)
3. 由于事务管理器要记录事务日志，必定会损耗一定的性能，并使得整个TCC事务时间拉长，建议采用redis的方式来记录事务日志。
4. tcc需要通过锁来确保数据的一致性，会加锁导致性能不高,只不过不是使用数据库的锁
   1. 依靠业务标识、单据状态、冻结标记、订单占用记录做占位锁定，属于业务层逻辑管控




#### 2-7 基于本地消息的最终一致性方案

- 本地消息表方案：比tcc方案性能高，并发更高，虽然一致性方案没有TCC那么强实时一致性，但可以保证最终一致性

- 核心逻辑：借助本地事务，同步完成业务数据操作与消息记录入库；再由定时任务把消息投递到**消息队列**，消费端处理业务并回执确认，生产者收到成功回执后，再清理本地消息表数据。即便消费服务宕机未处理，消息也会留存队列，等待下次重试消费。

- 本地消息表能解决以下问题：
1. 解决业务操作与消息发送原子性不一致问题
业务入库、发消息分属不同操作，常规写法极易出现：业务数据存库成功，消息却因网络、服务异常发送失败；或是消息发出去了，本地业务事务回滚。本地消息表借助本地数据库事务，把业务数据新增修改、消息记录写入绑定在一起，要么全都成功，要么全部回滚，从源头杜绝一边成功一边失败的情况。
2. 解决服务宕机、网络抖动导致的消息丢失
生产者服务突然宕机：消息已经存入本地数据表，重启后定时任务会自动补发，不会凭空丢失
网络瞬间中断：消息不会直接丢弃，留存表中等待网络恢复后重试投递
消费服务中途挂掉：未完成处理的消息留在队列，服务重启后继续消费，不会遗漏业务

![alt text](image-1.png)
- 启动本地服务会同时启动一个定时服务，定时服务会定时扫描本地消息表日志，表里会记录当初发送消息时有没有成功，记录mq的发送情况，下次服务重启时定时服务启动依然根据本地消息表进行重试发送，-----最终达到最终一致性
  - 隐含的问题：但消费端必须解决消息重复发送的问题，防止重复消费消息，增加好多异常积分，
- 总结:上面方式是一种非常经典的实现，基本避免了分布式事务，实现了“最终一致性”，但是，关系型数据库的吞吐量和性能方面存在瓶颈，频繁的重试读写消息会给数据库造成压力。所以，在真正的高并发场景下，该方案也会有瓶颈和限制的。

#### 2-8 基于可靠消息的最终一致性方案

如果生产者服务宕机、定时任务程序异常、数据库故障，就会出现消息丢失、重试失效、投递不及时等问题。同时轮询扫表也会带来一定的数据库 IO 压力。
- 为什么要抛弃「本地消息表」？
  - 本地业务 + 写消息表 + 发消息，无法做到原子性
    - 而RocketMQ 事务消息：半消息 + 本地事务 + 回查机制做到保证原子性
  - 可能出现：业务成功、消息表写入失败 / 消息表成功、发消息失败
  - 高并发下性能差，还要自己做重试、补偿、定时任务
    - 本地消息表：
      - 频繁读写消息表
      - 定时任务扫表
      - 锁、重试、IO 消耗大
    - RocketMQ 事务消息：
      - 无锁，异步化
      - 支持事务和回滚
        - 明确事务回滚的概念：回滚的是RocketMQ 里MQ发送方时的 “半消息”，是MQ消息half状态的回滚，并不能解决消费消息时执行失败时的反向回滚，不能反向通知MQ发送方让关联的本地事务进行回滚，如订单表创建回滚
      - MQ 天然高吞吐
      - 高并发场景性能差几倍到几十倍。
- 为了解决本地消息表消息不可靠、依赖本地轮询的缺陷，业界进一步优化出了 2-8 基于可靠消息的最终一致性方案，全程依托 MQ 中间件托管消息状态，彻底脱离本地定时任务兜底，可靠性更强、架构更标准。保证两件事必须同时成功 / 同时失败：本地数据库业务执行成功 和 消息一定发送到 MQ，被消费者消费

![alt text](image-2.png)

- rocketMQ给我们实现了这套方案，RocketMQ 是阿里开源的分布式消息中间件（MQ），它自带「事务消息」能力，正好解决你前面说的「本地消息表不可靠」的问题。
  - rabbitmq也有事务但是没有rocketmq的事务消息能力完善
  - 明确事务回滚的作用范围，上面已经讲了
- 最终一致性解决的是：本地发送出去的消息是可靠的，不会存在发出去后，本地失败，或者本地执行完了，消息没有发送出去
- 这依然有风险：1. 由于网络抖动，MQ发送方发送commit请求时失败了，---- 也有解法：rocketmq提供了一个回查机制，rabbitmqq就没有
  - 只要下次一启动rocketmq会定时向MQ发送方查询一下你这个消息迟迟状态未变，你到底是要commit还是要rollback，MQ发送方检查下本地数据库的实际业务情况，告诉MQ，如果commit就继续commit这个消息，否则就丢弃这个消息，MQ订阅方无感知
  - 支持高并发，因为它并没有锁的存在
- 需要额外维护rocketMQ服务的稳定性，它也支持分布式

   1. 一份业务数据，同步推送多个下游系统。一份消息多方订阅消费，高效完成数据同步、统计、推送等业务。

#### 2-2 rocketmq的基本概念

首先它是个分布式的消息队列框架

博客教程参考：https://www.seven97.top/microservices/message-queue/rocketMQ-basic.html#%E5%90%84%E8%A7%92%E8%89%B2%E4%BB%8B%E7%BB%8D

1. Producer:消息的发送者;举例:发信者
2. Consumer:消息接收者;举例:收信者
3. Broker:暂存和传输消息;举例:邮局
   1. 这是真正存储消息数据的地方
4. NameServer:管理Broker;举例:各个邮局的管理机构
   1. 相当于微服务中的注册中心consul，主要是管理多个broker之间的配置信息
5. Topic:区分消息的种类;一个发送者可以发送消息给一个或者多个Topic;一个消息的接收者可以订阅一个或者多个Topic消息
   1. Topic相当于公众号，每个人可以订阅不同的公众号，你可以向多个公众号发消息
6. MessageQueue:队列相当于是Topic的分区;用于并行发送和接收消息
   1. 队列（Queue/Partition）归属 Topic，是 Topic 内部的物理存储分区；
   2. 所有发送到该 Topic 的消息，最终都会落地到它下属的队列中排队存储
   3. Topic：逻辑分类，用来区分业务（如订单、日志、支付）；
   4. 队列：物理存储 & 读写单元，Topic 被拆分成多个队列做分片，实现并发与负载
   5. 一个 Topic 至少包含1 个队列，默认会配置多个；消息不会脱离队列单独存在。
   6. 生产者发消息时，MQ 按路由规则（轮询、哈希等）把消息分发到 Topic 下不同队列；消费者也是按队列维度拉取 / 推送数据。
7. 消费者组：队列 → 负责存消息，Group → 负责派人来消费它们之间：至于MQ如何自动完成topic内部多个队列与每个组内消费者的动态分配，这个工作分配关系即消息队列的负载均衡
   1. 负载均衡：把 Topic 下面的 N 个队列，平均分配给同一个 Group 里的 M 个消费者，负载均衡是对每个group纬度下的消费者进行负载均衡，不同 Group 各自独立做负载均衡
      1. 例：Group A、Group B 订阅同一个 Topic，两套分配逻辑互不影响
   2. 形象比喻：
      1. Topic = 一个快递站
      2. 队列 = 快递站里的 4 个货架（负责存放包裹）
      3. Group = 一个派送团队（比如：美团团队），同一个派送团队内的所有消费者进行负载均衡式地平分 Topic 下的所有队列，其他派送团队互不影响
      4. 消费者 = 团队里的派送员（小哥）

#### 2-3 roccketmq的消息类型

一、按照发送的特点分类:
1. 同步发送
2. 异步发送
3. 单向发送

![alt text](image-9.png)

二、按照使用功能特点分类:
1. 普通消息(订阅)
   1. 普通消息是我们在业务开发中用到的最多的消息类型，生产者需要关注消息发送成功即可，消费者消费到消息即可，不需要保证消息的顺序，所以消息可以大规模并发地发送和消费，吞吐量很高，适合大部分场景。
2. 顺序消息
   1. 消息有序指的是可以按照消息的发送顺序来消费(FIFO)。RocketMQ可以严格的保证消息有序，可以分为分区有序或者全局有序。顺序消费的原理解析，在默认的情况下消息发送会采取Round Robin轮询方式把消息发送到不同的queue(分区队列)；而消费消息的时候从多个queue上拉取消息，这种情况发送和消费是不能保证顺序。但是如果控制发送的顺序消息只依次发送到同一个queue中，消费的时候只从这个queue上依次拉取，则就保证了顺序。当发送和消费参与的queue只有一个，则是全局有序；如果多个queue参与，则为分区有序，即相对每个queue，消息都是有序的
   2. 顺序消息分为分区顺序消息和全局顺序消息，全局顺序消息比较容易理解，也就是哪条消息先进入，哪条消息就会先被消费，符合我们的FIFO，很多时候全局消息的实现代价很大，所以就出现了分区顺序消息。分区顺序消息的概念可以如下图所示:
   3. 我们通过对消息的key，进行hash，相同hash的消息会被分配到同一个分区里面，当然如果要做全局顺序消息，我们的分区只需要一个即可，所以全局顺序消息的代价是比较大的。
   4. 这种做法很明显就是并发不高，很多时候除非明确要顺序才会使用顺序消息
3. 延时消息：它主要用来做订单超时，库存归还的
   1. 延迟的机制是在服务端实现的，也就是Broker收到了消息，但是经过一段时间以后才发送服务器按照1-N定义了如下级别:"1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h";若要发送定时消息，在应用层初始化Message消息对象之后，调用Message.setDelayTimeLevelint level方法来设置延迟级别，按照序列取相应的延迟级别，例如level=2，则延迟为5s
      1. 实现原理:
         a. 发送消息的时候如果消息设置了`DelayTimelevel`,那么该消息会被丢到`ScheduleMessageService.SCHEDULE_TOPIC`这个Topic里面
         b. 根据DelayTimeLevel选择对应的queue
         c. 再把真实的topic和queue信息封装起来，set到msg里面
         d. 然后每个SCHEDULE_TOPIC_xxXx的每个DelayTimelevelQueue，有定时任务去刷新，是否有待投递的消息
         e. 每10s定时持久化发送进度
4. 事务消息：完全就是前面讲的基于可靠消息的最终一致性方案中的那个流程

##### 回查细节
- 只要下面任意一种情况发生，MQ 就会启动定时回查：
  - ExecuteLocalTransaction 返回了 UnknowState→ 明确告诉 MQ：我不知道成没成功，你等下问我。
  - ExecuteLocalTransaction 执行超时→ 执行太久没返回。
  - ExecuteLocalTransaction 内部崩溃、宕机、重启→ 根本没返回任何状态。
  - 提交 / 回查请求网络丢包→ MQ 没收到结果。
  - 只要 MQ 没收到明确的 Commit / Rollback→ 立即触发回查机制！
- 就算回查时服务挂了，下次服务重启，回查仍然会继续！为什么？因为 half 消息是持久化存在 MQ 里的！
  - 流程：
    - 发 half → 持久化
    - 服务执行本地事务时直接宕机
    - MQ 回查发现服务不可达
    - 服务重启恢复
    - MQ 继续回查
    - 服务查询 DB → 返回事务状态
    - 最终一致！
- 回查的三个返回值意义（必须记住）
  - CommitMessageState
    - 本地事务成功 → 消息投递
  - RollbackMessageState
    - 本地事务失败 → 消息删除
  - UnknowState
    - 还在执行中 → MQ 会继续回查
- 事务回查的作用：
  - 解决宕机、丢包、超时、异常
  - 保证本地事务和消息最终一致
  - 是分布式事务最可靠的兜底机制

#### 3-4 基于可靠消息最终一致性方案在库存服务上的不足
- 可靠消息方案只解决一件事：生产者 100% 能把消息发出去。
  - 本地事务成功 → 消息一定发
  - 本地事务失败 → 消息一定不发
  - 宕机、重启、丢包 → 靠回查兜底
  - → 保证消息可靠，不丢、不多、不乱
- 但它不保证消费者一定能处理成功
  - 消费者可能：
    - 宕机
    - 异常
    - 超时
    - 业务失败（比如库存不足）
  - 可靠消息不管消费者，只管生产者。
- 先要确定基于可靠消息的事务回滚概念：
  - Rollback即回滚的对象：RocketMQ 里的 “半消息”，只是回滚半消息，方案中并没支持回滚本地数据库事务
- 基于可靠消息方案本质上是解决了一个问题：**只是可靠消息的确保消息可靠**，但是无法确保你消费者能正确消费你这个消息。所以消费者应该自己确保你的消费逻辑没问题（包括服务正常或宕机等）， 所以它最终效果是建立在消费者代码和服务可靠的基础之上的。
  - 普通消费者服务一般问题不大：1. 宕机， 2. 代码有bug，解决了这些问题服务重启只要还没消费确认就能重新消费消息
  - 但是库存服务不太一样
    - 库存资源是有限的，有可能库存不足，不是无上限的，库存不足即失败还得反喂本地事务进行回滚，不像其他普通服务那样，只消费你消息就行，消费逻辑处不处理成功跟发送方没啥关系了，只管能接收到消息就行
    - TCC更适合库存服务这种情况，但是TCC没有解决方案自己实现又麻烦，TCC的并发又没有这个高
  - 我们可以折中一下，在可靠性方案基础上稍微改造一下

#### 3-15 rocketmq中shutdown造成的坑


- 订单接口文件`order_srv/handler/order.go`中，为什么我创建订单时的NewProducer实例p的shutdown，会影响超时发送时实例的shutdown，因为他们内部存储的key是以当前进程id为key，会冲突
  - RocketMQ 客户端内部，默认用【进程 ID】作为唯一标识存储 producer/consumer 实例
  - 同一个进程里创建多个 Producer → 实例互相覆盖、互相干扰！
  - 你 shutdown 其中一个 → 另一个也直接挂掉、无法发送消息！
  - 重点：应该使用全局单例
- 解决：
  - 使用时不要在一个进程中使用多个新producer实例，和随意进行shutdown，shutdown应该放在进程退出信号中
    - 推荐用法：全局单例 Producer：一个进程，只创建一个 Producer，到处复用，不要每次发消息都 New！
  - 创建NewProducer实例传入options参数指定
    - `rocketmq.NewProducer(producer.WithNameServer(ns),producer.WithInstanceName("inventory-producer-2"), // 不同名字)`


### 4章 实现接口的幂等性


#### 4-1 什么是服务雪崩

服务雪崩是非常严重的系统设计问题，

服务雪崩：微服务架构中，一个下游服务故障 / 响应变慢，故障顺着调用链向上扩散，逐级拖垮所有上游服务，最终整个链路、甚至整套系统全部不可用的连锁故障现象。

简单理解：一环崩，环环崩，全线瘫痪。


首要需要解决的是：超时机制

#### 4-2 超时、重试和幂等性机制的关系

他们之间是递进的关系，为什么要超时，因为防止服务雪崩的发生，为什么要重试，因为有超时机制，为什么要幂等，因为有重试请求

- 超时
  - timeout是为了保护服务，避免consumer服务因为provider 响应慢而也变得响应很慢，这样consumer可以尽量保持原有的性能。
- 重试
  - 如果provider只是偶尔抖动，那么超时后直接放弃，不做后续处理，就会导致当前请求错误，也会带来业务方面的损失。对于这种偶尔抖动，可以在超时后重试一下，重试如果正常返回了，那么这次请求就被挽教了，能够正常给前端返回数据，只不过比原来响应慢一点。重试可以考虑切换一台机器来进行调用，因为原来机器可能由于临时负载高而性能下降，重试会更加剧其性能问题，而换一台机器，得到更快返回的概率也更大一些。
- 幂等
  - 举例用户服务api层向用户服务的srv层发送写入用户的请求，如果网络抖动，如果有重试，又接收到了同样的数据过来，结果一次性注册2个客户
  - 因为重试机制，调用多次，数据库中只能正常显示一份，这就幂等了

#### 4-3 哪些情况下需要考虑幂等性问题

1. http请求的类型，更关心post和put方法
   1. get方法不需要
   2. post方法比较常见，会有幂等性问题
   3. put请求：不一定实现幂等性
      1. 如更改商品的价格改为200元，它重试的话，并没有影响 --- 这种情况没有幂等性问题
      2. 如购物车的商品数量，点击+1，就加1个数量，这时如果重试了  --- 这种情况就有幂等性问题
   4. delete操作：一般不具备幂等性要求，因为多次调用都是删除同一个数据

#### 4-5 常用的幂等性解决方案

1. 唯一索引，防止新增脏数据
   1. 比如: 新建用户的时候将手机号码设置为唯一索引，那么即使你重试，也只会新建一个用户，不会因为重试导致当前用户注册了两个用户
      1. 利用数据库的唯一索引机制
   2. 要点: 唯一索引或唯一组合索引来防止新增数据存在脏数据(当表存在唯一索引，并发时新增报错时，再查询一次就可以了，数据应该已经存在了，返回结果即可)
2. token机制：
   1. 思路：前端自己生成一个唯一 ID（UUID），提交时带上；后端用这个 ID借助redis等缓存库 做幂等 key。不用 “先向后端申请 Token”，前端本地生成即可。
   2. 不用前端额外请求拿 Token（省一次网络请求）
      1. 后端只加几行代码
      2. 能防：连点、网络重试、Nginx 重试
3. 悲观锁
4. 乐观锁
5. 分布式锁
6. select+insert
   1. 并发不高的后台系统，或者一些任务JOB，为了支持幂等，支持重复执行，简单的处理方法是，先查询下一些关键数据，判断是否已经执行过，在进行业务处理，就可以了
   2. 见`jieduan3-0-1shixian-weifuwu-kuangjia/mxshop_srvs/inventory_srv/handler/inventory.go`的Sell方法中
   3. 先查询StockSellDetail表，然后确定这个订单是否已经扣减过库存了，已经扣减过了就别扣减了
   4. 并发时候依然会有漏洞， 同一个时刻发送了重复了多次此时发现都没扣减过--- 这个最终解法还是使用锁，分布式锁
      1. 注意: 核心高并发流程不要用这种方法，有漏洞
7. 对外 API 接口幂等性---- 跟唯一索引类似
   1. 约定请求字段要求调用方必传两个参数：
      1. source：请求来源 / 商户标识，区分不同调用方
      2. seq：调用方本地唯一业务流水号，同一来源下永不重复
   2. 数据库层防重（关键）
      1. 在服务端数据表，对 source + seq 建立联合唯一索引。利用数据库唯一性约束，从底层拦截重复请求，并发场景下也只会执行一次业务逻辑。
   3. 标准执行流程（幂等友好）
      1. 接收请求，取出 source 和 seq；
      2. 优先根据这两个字段查询数据库，判断该请求是否已处理；
      3. 已处理：直接返回历史执行结果，不再执行业务逻辑；
      4. 未处理：执行业务逻辑，同时写入包含 source+seq 的记录，再返回响应。
   4. 注意：禁止不做前置查询直接执行业务，否则重复请求会触发索引冲突报错，同时业务已实际执行，造成数据与返回结果不一致。
8. 网关层统一做幂等性
   1. 在 Nginx / Gateway 层 做统一防重，业务代码完全不用改。
   2. 做法
      1. 网关拦截 POST 写请求
      2. 按 用户ID + 请求URL + body哈希 做 key
      3. Redis 记录：5 分钟内同一 key 只放行一次
   3. 优点
      1. 业务代码一行不改
      2. 统一拦截所有重复提交
      3. 防爬虫、防恶意重试
   4. 缺点
      1. 要改网关配置
      2. body 哈希有一点点性能损耗
   5. 适合：微服务、大型系统、不想改老代码。

### 1章 链路追踪系统技术概述

#### 1-1 为什么我们需要链路追踪？

微服务调用链长、节点多、故障难定位、性能难分析，链路追踪用来串联全链路请求，快速排障、优化性能、梳理调用关系。

- 单体应用：所有代码在一个进程，出问题看日志、查堆栈即可，调用链路简单，无需额外追踪。
- 微服务架构（核心痛点）：一次前端请求，会串行 / 并行调用网关→服务 A→服务 B→服务 C→数据库 / 缓存 / 消息队列，跨多节点、多进程、多机器：
  - 报错不知道哪一个服务 / 哪一段代码出问题；
  - 接口慢，分不清是网络、下游服务、DB 还是缓存拖慢整体耗时；
  - 调用关系复杂，人工梳理依赖、调用顺序成本极高；
  - 出现超时、丢请求、数据不一致，无法还原完整请求路径。

#### 1-4 opentracing的tracer、span、spanContext
OpenTracing 标准的3大核心概念

1. Tracer（追踪管理器）
   1. 作用：整个链路追踪的入口、管理员。
   2. 功能：
      1. 创建 Span
      2. 注入（Inject）上下文到请求头
      3. 提取（Extract）上游传递的上下文
      4. 管理采样、上报数据
   3. 一句话：Tracer = 链路追踪的总控制器，负责创建、传递、结束所有 Span。
   4. 链路追踪的管理者，负责创建、启动、结束 Span。
2. Span（一段调用 / 一个工作单元）
   1. 作用：代表链路中 “一次操作”。比如：
      1. 一次 HTTP 请求
      2. 一次 gRPC 调用
      3. 一次 DB 查询
      4. 一次函数执行
   2. 一个 Span 包含：
      1. SpanID
      2. TraceID
      3. 开始时间、结束时间
      4. 标签（tags：服务名、方法、状态码）
      5. 日志（logs：错误信息、参数）
      6. 父 SpanID（形成父子关系）
   3. 一句话：Span = 一次调用 / 一段耗时的记录。
   4. 代表一次调用 / 一段逻辑，记录耗时、状态、日志、标签。
3. SpanContext（链路上下文）
   1. 作用：在服务之间传递 Trace 信息。它里面只存 3 个东西：
      1. TraceID
      2. SpanID
      3. Baggage（透传字段）
   2. 它的核心功能：
      1. 跨服务透传 TraceID、SpanID
      2. 服务 A → 服务 B 必须靠它才能把链路串起来
      3. 不存储耗时、不存储标签，只传标识
   3. 一句话：SpanContext = 链路的 “身份证”，在服务之间传递，把整条链路串起来。
   4. 链路的上下文，存储 TraceID、SpanID，用于跨服务传递，把多个服务的调用串联成一条完整链路。

##### 重点细节
- TraceID（全局追踪 ID
  - 一条请求链路，只有一个 TraceID，从网关 → 订单服务 → 库存服务，全程不变
  - 作用：把所有服务的日志、Span 串成一条完整链路
  - Tracer：是一个 “工厂 / 管理器”，负责创建 Span、解析上下文、发送数据到 Jaeger Agent。
  - 正确理解：一个请求一个 TraceID，但 Tracer 全局唯一
  - TraceID：是根 Span创建时才生成的（整条链一个）。
- SpanID（片段 ID）
  - 每一个 Span 都有一个唯一的 SpanID
  - 父 Span、子 Span、孙子 Span，全都不一样
  - 作用：标识链路里的每一步操作
- span细节
  - Span 必须记录开始时间 + 结束时间
    - 靠 StartSpan() 和 Finish() 计算耗时。
  - 父子关系靠 ParentSpanID 维系，子 Span 里保存父 Span 的 ID。
  - Span 三大组成细节
    - Tags（标签，KV 结构）
      - 作用：查询、过滤、搜索
      - `span.SetTag("user_id", 1001)`
    - Logs / Events（日志，事件）
      - 作用：记录事件、错误、关键信息
      - `span.LogKV("event", "扣减库存成功")`
    - Baggage（透传字段）
      - 跨服务全局透传的 KV全链路传递
      - 如也可以设置和获取SpanContext（链路上下文）中的`span.SetBaggageItem("user_id", "1001")`



- TraceID = 链路身份证
  - 用来在 Jaeger 里搜索一整条请求。
- SpanID = 每一步的编号
  - 用来构成父子关系、调用顺序、耗时统计。

package middlewares

import (
	"mxshop-api/goods-web/global"

	"github.com/gin-gonic/gin"
	"github.com/opentracing/opentracing-go"
)

// Trace 全局唯一的 Tracer，在 main 中初始化一次
func Trace() gin.HandlerFunc {
	return func(ctx *gin.Context) {
		// 从全局获取 tracer（只初始化一次）
		tracer := opentracing.GlobalTracer()

		// 开启一个 span，作为当前请求的 根span ---- 这时才会生成traceid
		span := tracer.StartSpan(ctx.Request.URL.Path)
		defer span.Finish()

		// 将 tracer & 父span 存入 gin context
		ctx.Set("tracer", tracer)
		ctx.Set("parentSpan", span)

		ctx.Next()
	}
}

// main.go 文件：全局初始化一次
func InitJaeger() {
	cfg := jaegercfg.Configuration{
		Sampler: &jaegercfg.SamplerConfig{
			Type:  jaeger.SamplerTypeConst,
			Param: 1,
		},
		Reporter: &jaegercfg.ReporterConfig{
			LogSpans:           true,
			LocalAgentHostPort: fmt.Sprintf("%s:%d", global.ServerConfig.JaegerInfo.Host, global.ServerConfig.JaegerInfo.Port),
		},
		ServiceName: global.ServerConfig.JaegerInfo.Name,
	}

	tracer, closer, err := cfg.NewTracer(jaegercfg.Logger(jaeger.StdLogger))
	if err != nil {
		panic(err)
	}

	// 只设置一次
	opentracing.SetGlobalTracer(tracer)

	// 程序退出时关闭一次
	global.JaegerCloser = closer
}

#### 3-1 什么是限流熔断和降级？

![alt text](image-15.png)

高并发场景下如何来防止服务的雪崩？

![alt text](image-16.png)

1. 限流
   1. 流量过来2k但是我的服务能力只有1k，所以这个时候多出来的流量怎么办:1.拒绝2.排队等待用户体验不太好:当前访问用户过多，请稍后重试和你的服务直接挂了
2. 熔断
   1. 比如A服务访问B服务，这个时候B服务很慢-B服务压力过大，导致了出现了不少请求错误，调用方很容易出现一个问题:每次调用都超时2k，结果这个时候数据库出现了问题，超时重试·网络2k的流量突然变成了3k这让原本就满负荷的b服务雪上加霜，如果这个时候调用方有一种机制:比如说
      1. 发现了大部分请求很慢-50%请求都很慢
      2. 发现我的请求有50%都错误了
      3. 错误数量很多，比如1s出现了20个错误
   2. 熔断-1.保险丝2.股市熔断1
3. 降级
   1. 用户体验降级了-原本是访问流畅，下单流畅->当前访问用户过多，请稍后重试
   2. 上面的限流和熔断之后，最终都会导致服务降级

#### 3-4 sentinel的预热功能

- 测试代码课件见：`jieduan5-分布式系统核心、微服务的部署/sentinel_test/flow/warm_up`目录

**预热它到底解决什么问题？**：
- 场景：
  - 服务刚启动
  - 缓存是空的
  - JIT / 数据库连接 / 本地缓存都没初始化
  - 流量突然直接冲到 100 QPS
  - → 直接压死！
- WarmUp 做的事
  - 不让流量瞬间冲满！
  - 让流量慢慢涨，给系统预热时间！
- 比如：
  - 0 秒：允许 QPS = 30
  - 3 秒：允许 QPS = 50
  - 6 秒：允许 QPS = 80
  - 10 秒：才放开到 100 QPS
  - 像开车一样，慢慢踩油门，不是直接飙到最大速度！

#### 3-5 sentinel的Throttling配置策略

- 直接拒绝
  - 如设置qps10，每 100ms 放 1 个请求，每秒就处理10个，测试循环遍历12个时，时前10个都通过，第11个拒绝
- 匀速通过：请求扎堆来时，不直接拒绝，而是让它等够qps的时间间隔；等待超时才拒绝
  - 如设置10，每个请求等待间隔就是100ms，结果，第一个通过，后面全部拒绝了
  - 因为循环遍历12个发送，间隔都小于100ms，所以全部拒绝了

#### 3-6 sentinel的熔断机制

- 见`jieduan5-分布式系统核心、微服务的部署/sentinel_test/breaker`目录
- 官方文档：https://sentinelguard.io/zh-cn/docs/golang/circuit-breaking.html
    - 详细概念看文档吧，这里笔记为辅

- 熔断器模型：Sentinel 完全遵循经典熔断器状态机
- 1）CLOSED 关闭（正常）
  - 请求全部放行
  - 持续统计：响应时间、异常比例、异常数
  - 条件：指标没到阈值 → 一直 Closed
- 2）OPEN 打开（熔断）
  - 指标超过阈值 → 切到 Open
  - 所有请求直接拒绝，走降级（Fallback）
  - 持续 timeWindow（熔断时长，如 10s）
- 3）HALF-OPEN 半开（探测）
  - 熔断时间到 → 进入 Half-Open
  - 只放 1 个探测请求，其他仍拒绝
  - 探测成功 → **切回 Closed（恢复正常）**
  - 探测失败 → **切回 Open（继续熔断）**


- Sentinel 提供以下几种熔断策略：
  - 慢调用比例 (SLOW_REQUEST_RATIO)：选择以慢调用比例作为阈值，需要设置允许的慢调用 RT（即最大的响应时间），请求的响应时间大于该值则统计为慢调用。当单位统计时长（statIntervalMs）内请求数目大于设置的最小请求数目，并且慢调用的比例大于阈值，则接下来的熔断时长内请求会自动被熔断。经过熔断时长后熔断器会进入探测恢复状态（HALF-OPEN 状态），若接下来的一个请求响应时间小于设置的慢调用 RT 则结束熔断，若大于设置的慢调用 RT 则会再次被熔断。
  - 异常比例 (ERROR_RATIO)：当单位统计时长（statIntervalMs）内请求数目大于设置的最小请求数目，并且异常的比例大于阈值，则接下来的熔断时长内请求会自动被熔断。经过熔断时长后熔断器会进入探测恢复状态（HALF-OPEN 状态），若接下来的一个请求成功完成（没有错误）则结束熔断，否则会再次被熔断。异常比率的阈值范围是 [0.0, 1.0]，代表 0% - 100%。
  - 异常数 (ERROR_COUNT)：当单位统计时长内的异常数目超过阈值之后会自动进行熔断。经过熔断时长后熔断器会进入探测恢复状态（HALF-OPEN 状态），若接下来的一个请求成功完成（没有错误）则结束熔断，否则会再次被熔断。
- Sentinel 熔断器的三种熔断策略都支持静默期 (规则中通过MinRequestAmount字段表示)。静默期是指一个最小的静默请求数
  - 在统计窗口内，请求数量必须 ≥ 这个值，熔断器才开始判断是否熔断，防止 “少量请求波动” 误触发熔断！


- 熔断器一般用于应用对外部资源访问时的保护措施。这里简单描述一些场景：
  - 分布式系统中降级：假设存在应用A需要调用应用B的接口(特别是一些对接外部公司或者业务的接口时候)，那么一般用于A调用B的接口时的防护；
  - 数据库慢调用的防护： 假设应用需要读/写数据库，但是该读写SQL存在潜在慢SQL的可能性，那么可以对该读写接口做防护，当接口不稳定时候(存在慢SQL)，那么基于熔断器做降级。
  - 也可以是应用中任意弱依赖接口做降级防护（即自动降级后不影响业务核心链路）。

#### 2-2 基本的路由转发配置即动态路由

- konga中最重要的2个概念：Service（服务）和 Route（路由）
- services:

    - 一个service可以有多个route
- Routes:Route 路由相当于nginx 配置中的location，Route实体定义匹配客户端请求的规则.每个路由都与一个服务相关联,而服务可能有多个与之相关联的路由.每一个匹配给定路线的请求都将被提交给它的相关服务.
- Service（服务）
  - 代表一个后端服务（比如 user-service）
  - 核心是：我要代理到哪里去
  - 配置方式二选一：
  - 简单：直接写死 host + port（单实例）

  - 是一个虚拟主机名，下面挂一堆 Target（IP: 端口），一个service对应一个upstream，一个upstream对应多个某服务的实例
  - 专门负责：负载均衡、健康检查、故障转移
  - 例如：
  - Upstream 名叫 user-upstream
  - Target：192.168.1.10:8080、192.168.1.11:8080
  - 可以理解成：一个负载均衡器，管理一群真实节点。
- 路由和服务的组合(以及它们之间的关注点分离)提供了一种强大的路由机制，可以在Kong中定义细粒度的入口点,从而引导您的访问到不同upstream服务.

#### 3-x 概念总结
**jekins部署模式：**
1. 单节点模式（单机）
   1. 所有功能：Web 界面、调度、拉代码、编译、打包、部署，全在一台机器上跑。
   2. 又叫：独立节点、单 Master。
2. 分布式模式（主从 Master-Agent）
   1. Master（主节点）：只做 “调度 + 界面 + 管理”，不干活（不编译打包）。
   2. Agent（从节点 / 代理节点）：专门干活：拉代码、编译、测试、打包、部署。
   3. 一个 Master → 可以挂 N 个 Agent（Linux/Windows/macOS/ 容器都行）。------ 相当于gitlab的runner机
3. 延伸（云原生）：K8s 动态 Agent（弹性主从）
   1. Master 对接 K8s，任务来了自动创建 Pod 当 Agent，跑完自动销毁，资源利用率最高。


```js
系统	      单机	                           主从 / 分布式	                                                  核心区别
Jenkins	一个节点：Web + 调度 + 构建	       Master（调度）+ Agent（构建）	                                   主从 =调度和执行分离
GitLab	一个节点：Web+DB + 仓库 + Runner	   1. 服务：Web 集群 + DB 主从 + Gitaly 集群2. CI：GitLab Runner 集群	服务是分层分布式；Runner 才是执行节点
```

#### 3-7 通过pipline实现持续集成

> pipline模式相比freestyle模式更高级一些门槛高一些。

- 之前安装git插件后，新建任务时，会自带一个流水线pipline出来，我们可以基于这个pipline，实现持续集成
- 把你之前在 Freestyle（自由风格）页面上点来点去的配置，变成一段代码，存进项目里，可版本管理、可复制、可复用。
- 用pipline流水线时，还需要再装一个插件：`pipline`插件

- Jenkins Pipeline 是Jenkins 自己实现的流水线插件，基于 Groovy 实现的 CI/CD 专用 DSL（领域专用语言）
- 主要分两种写法：
  - 1. Declarative Pipeline（声明式，推荐 99% 用这个）,像写配置文件一样（如travis.yml、gitlab-ci.yml）
  - 2. Scripted Pipeline（脚本式，老版本用）
- 优点
  - 1. 代码即配置
    - 构建流程写在代码里，提交到 Git，不会丢失。
  - 1. 可复用、可复制
    - 一个项目的流水线，复制到另一个项目直接用。
  - 1. 流程可视化
    - 每一步构建都能看到：
  - 拉代码→编译→扫描→打包→上传→部署
    - 哪一步错了一目了然。
  - 1. 支持复杂逻辑
    - 多环境、多服务器、并行构建、条件判断……
    - Freestyle 做不到的，Pipeline 都能做。
- 然后点击流水线任务里进去一行行配置，主要是配置流水线的脚本
```groovy
// 例如在pipeline的流水线配置项里选的Pipline script模式，直接在页面里书写配置的
pipeline {
    agent any
    stages {
        stage('拉代码') {
            steps {
               // git插件提供的方法
                git url: 'https://gitee.com/xxx/xxx.git', 
                    branch: 'main', 
                    credentialsId: '你的凭证ID'
            }
        }

        stage('编译') {
            steps {
                sh 'mvn clean package -DskipTests'
            }
        }

        stage('发布') {
            steps {
            // Publish Over SSH插件提供的方法
                sshPublisher(publishers: [sshPublisherDesc(
                    configName: '你的服务器名',
                    transfers: [sshTransfer(
                        sourceFiles: 'target/*.jar',
                        removePrefix: 'target',
                        remoteDirectory: '/opt/app'
                    )],
                    execCommand: '''
                        cd /opt/app
                        pkill -f app.jar
                        nohup java -jar app.jar > run.log 2>&1 &
                    '''
                )])
            }
        }
    }
}
// 这些插件对应的语法，可以直接方便地在jekins转化页面中，通过默认freestyle模式插件输入框输入，生成对应的脚本
```

#### 4-5 通过go build构建go微服务

1. 首先需要在jenkins的构建服务器上安装go环境，实现直接使用go能找到命令，中间需要设置linux的环境变量
2. jekins流水线执行时找不到go命令，只能在构建流水线中的shell脚本在显示设置下go的环境变量和环境准备工作，如开启go module模式
3. 然后先构建好目录结构，每个项目对应一个服务器上的目录下，然后必须打开目录文件权限
   1. 因为本地测试中发现直接执行打包好的可执行文件会报错：报找不到配置文件，所以额外给他创建一个目录，并复制配置文件过去，然后可执行文件在这个目录中执行就可以了

```shell
# shell脚本大致如下
# ======================
# 1. 配置 Go 语言环境变量
# ======================
echo "===== 配置 Go 环境变量 ====="
export GOROOT=/usr/local/go
export PATH=$PATH:$GOROOT/bin

# 验证是否生效
go version

# ======================
# 2. 配置 Go module 代理
# ======================
echo "===== 配置 Go module 代理 ====="
go env -w GO111MODULE=on
go env -w GOPROXY=https://goproxy.io,direct

# ======================
# 3. 准备目录结构 & 赋权
# ======================
echo "===== 准备目录结构 ====="
chmod 777 -R goods_srv/
mkdir -vp goods-web/target/goods-web

# ======================
# 4. 复制配置文件 ---- 因为本地测试中发现直接执行打包好的可执行文件会报错：报找不到配置文件，所以额外给他创建一个目录，并复制配置文件过去，然后可执行文件在这个目录中执行就可以了
# ======================
echo "===== 复制配置文件 ====="
cp goods-web/config-pro.yaml goods-web/target/goods-web/config-pro.yaml

# ======================
# 5. 进入目录编译
# ======================
echo "===== 开始编译 Go 项目 ====="
cd goods-web
go build -o target/goods_web_main
```

#### 4-6 发布go项目到远程服务器上

- 细节注意点

### 5章 课程总结和未来方向

#### 微服务架构演进路线

单体应用 → 微服务 → 服务网格(Service Mesh) → 云原生(Serverless)

1. 单体应用（All in one）
   1. 所有功能写在一个项目里
   2. 优点：简单、部署快
   3. 缺点：无法扩容、牵一发动全身、大项目卡死
2. 微服务（核心阶段）
   1. 按业务拆分成多个服务
   2. 独立开发、独立部署、独立数据库
   3. 技术栈：Go、Java、gRPC、HTTP、MQ、注册中心、配置中心
3. 服务网格 Service Mesh（基于 K8s）
   1. 把流量治理、限流、熔断、链路追踪从业务代码剥离
   2. 代表：Istio
   3. 特点：无侵入、统一治理、企业级高可用
4. 云原生 & Serverless
   1. 不用关心服务器
   2. 自动扩缩容、按调用收费、极致弹性
   3. 代表：阿里云函数计算、AWS Lambda
