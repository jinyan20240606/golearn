### 1章 protoc插件开发

#### 1-1 工程实践中如何更好的使用proto文件？

- 借鉴了在kratos这个微服务框架中对proto文件的灵活用法，我们参考学习它的灵活用法

1. proto文件可以用作http和rpc服务的生成标准写法
   1. 我写了一个gin的服务，我还要手动去维护api文档，手动去yapi上维护
   2. 当有了proto后，后期维护和迭代很简单， 改了任何代码你都可以直接生成api
   3. 可以直接将proto生成swagger文件，然后一键导入到yapi上，这样就可以直接在yapi上查看api文档了
   4. 定义即文档：.proto 文件本身就是最严谨、无歧义的接口文档。
   5. 代码即生成：通过 protoc 配合插件（如 protoc-gen-go-http），一键生成 HTTP 路由、请求/响应结构体、甚至 Swagger/OpenAPI 的 JSON 文件2。
   6. 无缝对接 YApi：生成的 OpenAPI 文档可以直接导入 YApi，完全省去了手动录入的繁琐。
2. 在kratos中对proto的依赖更加重， 可以用来定义一些错误码， 并生成go源码直接使用，kratos甚至将配置文件都给你映射成proto文件
   1. 错误码标准化：Kratos 允许在 .proto 中定义业务错误码（errors.proto），并自动生成 Go 的错误处理代码。这样全公司的微服务错误码都能保持统一规范，前端处理异常时也极其方便。
   2. 配置即代码：将配置文件映射为 Proto 结构体，然后自动生成Go 的配置对象的结构体，用来解析Nacos、Apollo 等配置中心的配置，直接校验解析配置，可以实现配置的热加载和强类型校验。再也不用担心读取配置时因为字段名写错而导致程序 panic2。
   3. 参数校验自动化：配合 proto-validate 插件，你甚至可以直接在 .proto 文件里写上校验规则（比如 string email = 1 [(validate.rules).string.email = true]），生成的代码会自动帮你拦截不合法的请求参数。
3. 业内很多框架都开始逐步接受将proto文件作为核心的标准去写一系列插件去自动生成代码
   1. proto validate
   2. go-zero更溜它自带的goctl，保姆式的框架 内部独创的api接口定义语言体系用来契约先行
   3. go-zero和kratos的是不同的设计理念，kratos是插拔式，用proto文件作为核心标准去生成代码
4. 现代微服务开发（尤其是 Go 语言生态）的核心范式转变——“契约先行”（Contract-First）
   1. 它们不再把 .proto 仅仅当作 gRPC 的接口定义，而是将其提升为整个微服务架构的“单一事实来源”（Single Source of Truth）

#### 1-2 protoc命令如何查询依赖的proto文件

- 示例代码见`jieduan7-效率工具开发/proto/api.proto`

1. 当在api.proto中 引入本地三方的proto依赖：`import "google/api/annotations.proto";`
   1. Protobuf 的 import 机制和 Go、Java 等语言的包管理不太一样，它不会自动去 $GOPATH 或 node_modules 里找依赖，而是完全依赖显式指定的搜索路径
   2. protoc 编译器在解析 .proto 文件时，默认只会去当前目录（.）下寻找被 import 的文件。当你引入了 google/api/annotations.proto 这种外部依赖时，protoc 根本不知道去哪里找它。
      1. 因此，必须通过 -I（或 --proto_path）参数，明确告诉 protoc 去哪里搜索这些依赖
      ```bash
        protoc -I . \
            -I ./third_party \
            --go_out=. \
            --go-grpc_out=. \
            api.proto
      ```
   3. IDE 的智能提示（飘红）和命令行的实际编译
      1. 默认需要在IDE编辑器插件中设置下三方proto目录，否则飘红报错，在命令行执行proto编译时，也需要手动指定三方proto目录范围
      2.  IDE 内置了一套独立的 Protobuf 解析引擎，用来提供代码跳转、自动补全等智能提示。它不会自动读取你命令行里的 -I 参数
2.  如何优雅地管理这些三方 proto 依赖？
    1.  用proto管理工具（类似于npm）：buf工具：https://blog.csdn.net/m0_57836225/article/details/149730913
    2. 完美解决了你刚才提到的“手动指定三方 proto 目录”以及“飘红报错”的痛点，把原本繁琐的 protoc 编译、依赖管理和代码生成流程全部自动化了



#### 1-3 protoc执行插件的原理
- 核心原理： protoc 主进程与插件子进程之间，通过 标准输入输出（stdin/stdout） 传递 Proto 序列化的二进制数据
- 执行四部曲：
  - 启动与传参：当你执行类似`protoc --go_out=. api.proto` 的命令时，protoc 会根据 `--xx_out` 参数（这里是 `go_out`）去寻找对应的插件
    - `--xx_out`: 规则就是xx就是插件名，对应一个可执行文件，下划线后面的是插件对应的参数
    - 默认查找：它会去系统的环境变量 $PATH 中，寻找名为 protoc-gen-xx 的可执行文件（比如 protoc-gen-go）1。
    - 显式指定：你也可以通过 --plugin=protoc-gen-xx=/path/to/your/plugin 参数，明确告诉 protoc 插件的具体位置1。
    - 找到插件后，protoc 主进程会通过 fork 和 exec 的方式，将这个插件作为一个独立的子进程启动起来
  - 插件处理：插件从 stdin 读取并反序列化请求，遍历消息/服务节点，通过模板或字符串拼接生成目标代码1。
  - 返回结果：插件将生成的代码文件名与内容封装成 CodeGeneratorResponse，序列化后写入自己的 stdout1。
  - 文件落地：protoc 捕获插件的 stdout，解析响应并将代码文件写入指定磁盘目录2。
- 核心避坑点：
  - 语言无关：只要能读写 stdin/stdout 并处理 Proto 序列化，任何语言（Go/Python/Rust等）都能写插件2。
  - 长度前缀（Length Prefix）：protoc 发送的数据不是裸的 PB 字节流，前面带有一个 Varint 编码的长度头（[4字节长度][实际PB数据]）。读取 stdin 时必须先读长度再读内容，否则必然解析失败

#### 1-4 proto文件中的package 和 go_package的作用区别

- 简单来说：package 是给 Proto 自己用的（Proto 命名空间），而 go_package 是专门给 Go 语言用的（Go 包名和路径）。
  - 编译后生成的文件不需要关心Proto 命名空间

假设你有一个 user.proto（作为被引用的基础文件）：
```go
// user.proto
syntax = "proto3";

// 定义了 user.v1 的 Proto 命名空间
package user.v1;
// 决定了生成的 Go 代码包名是 userv1
// 1参：指定生成的 .pb.go 文件在项目中的存放位置
// 2参：指定生成的 .pb.go 文件顶部的 package 声明。这决定了在 Go 代码中如何引用该文件中定义的类型（例如 userv1.User）
option go_package = "github.com/yourname/myproject/api/user;userv1";

message User {
  string name = 1;
  int32 age = 2;
}
```
现在，我们在同一个目录下新建一个 order.proto，并在里面引用 User：
```go
// order.proto
syntax = "proto3";

// 定义当前文件的 Proto 命名空间
package order.v1;
// 定义当前文件生成的 Go 代码包名
option go_package = "github.com/yourname/myproject/api/order;orderv1";

// 【关键点1：Proto 层面的引用】
// 必须显式 import 对应的 .proto 文件，否则 protoc 找不到 User 的定义
import "user.proto";

message Order {
  string order_id = 1;
  
  // 【关键点2：Proto 命名空间的使用】
  // 因为 user.proto 里声明了 package user.v1;
  // 所以在这里引用 User 消息时，必须带上它的 Proto 包名前缀：user.v1.User
  user.v1.User buyer = 2; 
}
```
当你把这两个文件都编译成 Go 代码后，生成的 order.pb.go 会是这样的：
```go
// order.pb.go
// Code generated by protoc-gen-go. DO NOT EDIT.

// 由 order.proto 的 go_package 决定
package orderv1

import (
    // 【关键点3：Go 语言层面的引用】
    // protoc 非常智能，它发现 Order 结构体用到了 User。
    // 于是它自动去 user.proto 的 option go_package 里找到了导入路径，
    // 并自动在 Go 文件里加上了这一行 import！
    userv1 "github.com/yourname/myproject/api/user"
)

type Order struct {
    OrderId string           protobuf:"bytes,1,opt,name=order_id,json=orderId,proto3" json:"order_id,omitempty"
    // 在 Go 结构体中，直接使用上面 import 进来的 userv1 包来调用 User 结构体
    Buyer     *userv1.User   protobuf:"bytes,2,opt,name=buyer,proto3" json:"buyer,omitempty"
}
```
通过这个完整的例子，你可以清晰地看到整个链路：

在 .proto 文件里（Proto 世界）：
   你想引用别的文件，必须先 import "xxx.proto"。引用具体的消息时，要遵守对方定义的 package 规则（比如 user.v1.User）。

在生成的 .pb.go 文件里（Go 世界）：
   protoc 编译器充当了完美的“翻译官”。它会自动把 Proto 的 import 翻译成 Go 的 import，把 Proto 的 package 前缀（user.v1）翻译成 Go 的包别名（userv1）。

所以在写业务代码调用时，你完全不需要关心 Proto 的 package，只需要像平时写 Go 代码一样：
```go
package main

import (
    "github.com/yourname/myproject/api/order" // 导入路径
    "github.com/yourname/myproject/api/user"  // 导入路径
)

// 真正使用：
order := &orderv1.Order{
    OrderId: "12345",
    Buyer: &userv1.User{
        Name: "张三",
        Age:  25,
    },
}
```
这就是 package 和 go_package 在跨文件引用时各司其职、互不干扰的完美体现！

#### 1-5 protoc 插件开发原理

- 插件 = 独立可执行程序（不是动态库、不是脚本、不是绑定）。
- protoc 把它当子进程跑，用 stdin/stdout 传结构化数据（protobuf 消息）。
- 命名规则：
  - 插件二进制必须叫：protoc-gen-xxx
  - 调用时：--xxx_out = 输出目录
- 二、protoc ↔ 插件 通信流程（最核心），整个过程只有 4 步，非常固定：
  - 1. protoc 先把所有 .proto 解析成 proto专属的AST
    - 解析语法、校验、resolve import
    - 生成一堆 FileDescriptorProto（就是 proto 的结构化表示）
  - 1. protoc 构造 CodeGeneratorRequest（发给插件的数据包）
    - 内容包括：
    - 所有 proto 文件的描述符（FileDescriptorProto）
    - 要生成哪些文件
    - 命令行参数（如 --go_out=paths=source_relative）
  - 1. protoc fork 子进程（插件），把 Request 写入插件 stdin
    - 二进制格式、带长度前缀
    - 完全是 进程间管道通信
  - 1. 插件处理后，返回 CodeGeneratorResponse 到 stdout
    - 包含：要生成的文件名 + 文件内容
    - 或返回 error
    - protoc 拿到后真正写文件到磁盘
- 三、通信协议（plugin.proto），这是插件世界的 “标准接口”，定义在 protobuf 源码里：
  - `message CodeGeneratorRequest {`
  - `message CodeGeneratorResponse {`
- 四、插件内部逻辑
  - 不管用 Go、C++、Python，逻辑永远是固定的 3 步：
    - 从stdin 读取 → 解码成 CodeGeneratorRequest
    - 遍历 FileDescriptorProto，生成你要的代码（Go/Java/ 自定义）
    - 把结果打包成 CodeGeneratorResponse → 写入 stdout
  - 在 Go 里，官方已经封装好 protogen 库，你只需要写生成逻辑：
- 示例代码见`jieduan7-效率工具开发/tools`
  - 我们要`go build 这个main.go文件`，进行自定义插件的构建，期间我们修改了protogenAPI的`protogen.Options{ParamFunc: flags.Set}.Run`源码让他内部写死读取一个指定proto文件，,方便进行自定义插件的测试调试
    - 具体源码修改见`1-6节`

##### proto概念扫盲

1. google/api/annotations.proto 该插件作用：扩展 Protobuf 的 option 语法
   1. Protobuf 原生自带的 option语法 只有这些：`option go_package = "...";`
      1. option = Protobuf 的 “配置项”，给 proto 文件 / 服务 / 方法 / 字段 加 “额外设置”
      2. Protobuf 原生只有 2 种 option
         1. 文件级：写在文件最上面，配置整个文件怎么生成代码
         2. 字段级：`string name = 1 [json_name = "username"];`
         ```go
            message User {
                string user_name = 1 [json_name = "username"];
            }
            //生成 Go 结构体时，自动变成：
            type User struct {
                UserName string `protobuf:"bytes,1,opt,name=user_name" json:"username,omitempty"`
            }
         ```
         3. 不支持方法级
   1. annotations插件就是给 Protobuf 扩展了「方法级 option」！，你 import annotations之后，protoc就认识了
      ```go
      option (google.api.http) = {
        get: "/user"
        };
      ```

      9. `bytes 包` = 用来操作「字节数组」的工具包
         1.  里面最最常用的，就是 bytes.Buffer
         2.  作用：一块可写、可读、可拼接的内存缓冲区
