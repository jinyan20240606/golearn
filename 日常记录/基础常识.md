# 基础常识

## 包积累

1. 日志框架
   1. fmt：基础打印工具，无日志管理能力，只适合临时调试；
   2. logrus：企业级日志框架，分级、持久化、结构化、全局统一配置，是后端服务标准方案
      1. 一般在项目中本地打印同时结合Graylog采集推送远端，无需登录服务器看日志，网页端检索、导出、告警
2. 日志分析系统主流架构套件
   1.   1. ELK
        Elasticsearch + Logstash + Kibana
        经典老式架构：
        应用日志 → Logstash（采集 + 清洗） → ES → Kibana
        Logstash：JVM 重型中间件，同时做采集、复杂清洗、转发
        缺点：Logstash 耗内存（启动 500MB+），不能部署在业务机器，只能独立集群
        2. EFK
        Elasticsearch + Fluentd/Fluent Bit + Kibana
        K8s 云原生主流方案，用 Fluent 全家桶替换 Logstash：
        应用容器 stdout → Fluent Bit（DaemonSet 节点采集） → ES
        Fluent Bit：纯 C 开发，内存仅 1~10MB，每个节点部署一个
        Fluentd：Ruby+C，40MB 左右，适合做中转清洗层
        3. ELFK（生产最标准架构，推荐）
        Elasticsearch + Logstash + Filebeat + Kibana
        分层解耦架构：
        业务机器 / 容器 → Filebeat（轻量采集） → Logstash（集中清洗） → ES → Kibana
        Filebeat：Go 写，30MB 以内，只负责本地日志采集、磁盘缓存、转发，不做复杂解析
        Logstash 独立集群，统一做日志脱敏、grok 解析、geoip、分流、过滤
        兼顾轻量采集与强清洗能力，互联网大厂标配
        4. Graylog（一体化平台，底层也套 ES）
        Graylog Server + ES + MongoDB + Kibana
        流程：应用直推 GELF / Filebeat 采集 → Graylog 服务 → ES
        Graylog 把「接收端 + Logstash 清洗 + Kibana 可视化 + 告警权限」打包成单一服务
        ES 只做底层存储，用户几乎不用直接操作 ES、Kibana
   2. 传统物理机、复杂日志解析、专业运维 → ELFK
   3. 纯 K8s 容器、海量节点、节省机器内存 → EFK(Fluent Bit)
   4. 小团队、不想维护多套组件、统一告警权限面板 → Graylog
   5. 单机小项目、快速测试、不想部署采集器 → 直推 Graylog 优于 elogrus 直推 ES
